## 四、各文档类型审查要点

### 4.1 设计文档审查 (design/*.md)

| 审查维度 | 重点 |
|------|------|
| 一致性 | 与 project.md 的目标是否对齐；与 config.yaml rules 是否冲突 |
| 完整性 | 是否覆盖了 Non-Goals 的边界；是否列出了替代方案 |
| 可行性 | 技术选型是否合理；依赖是否真实可用；工作量是否现实 |
| 准确性 | 性能指标是否有依据；竞品对比是否公正 |

### 4.2 数据模型审查 (data-structures.md)

| 审查维度 | 重点 |
|------|------|
| 一致性 | 字段命名是否跨模型统一；类型是否前后端一致 |
| 完整性 | 是否缺少关键时间戳；是否有必要的索引字段；边界值处理 |
| 可行性 | 存储估算是否合理；二进制格式解析复杂度 |
| 准确性 | 单位是否统一 (ms/s, km/h/m/s, m/ft)；坐标系是否明确 |

### 4.3 路线图审查 (roadmap.md, status.md)

| 审查维度 | 重点 |
|------|------|
| 一致性 | Phase 之间的依赖是否正确；与 design 文档的能力是否对应 |
| 完整性 | 是否有遗漏的关键里程碑；是否有验收标准 |
| 可行性 | Phase 工期是否合理；资源是否充足 |

### 4.4 变更提案审查 (changes/*/proposal.md)

| 审查维度 | 重点 |
|------|------|
| 一致性 | Why 与项目目标是否对齐；Impact 是否与其他 change 冲突 |
| 完整性 | Capabilities 是否完整列举；What Changes 是否有遗漏 |
| 可行性 | 变更范围是否可控 |

### 4.5 任务清单审查 (changes/*/tasks.md)

| 审查维度 | 重点 |
|------|------|
| 一致性 | 任务是否覆盖了 design 中的所有 Decisions |
| 完整性 | 是否有遗漏的集成/测试/文档任务 |
| 可行性 | 任务粒度是否合适（可独立验证） |

## 五、审查触发条件

| 触发场景 | 说明 |
|------|------|
| Worker 完成新文档 | Worker 主动声明审查范围 |
| Worker 修改已有文档 | Worker 说明修改内容和影响范围 |
| Monitor 发现潜在问题 | Monitor 可主动发起定向审查 |
| Phase 状态变更 | Phase 开始/完成时触发全量审查 |
